# Docsy Word 模板系统设计

更新时间：2026-07-16

## 目标

Docsy 模板系统用于把律所高频 Word 文书做成可重复填写的模板。它不是 Word 编辑器，也不是简单邮件合并工具。Word 负责排版和选区切分，Docsy 负责字段确认、数据填写、历史建议和批量生成。

核心流程：

```text
Word 中标黄可变文字
→ Docsy 扫描黄色标记
→ 用户确认字段类型、字段名和勾选组
→ 保存为 .docsytpl
→ 打开模板填表
→ 生成 docx
→ 可选用 Word/WPS/LibreOffice 转 PDF
```

## 架构边界

新模板模块必须独立于旧模板系统。旧的字典、手写推理规则、模板归档和 Docsy 内部文本选区定位不再复用。

可复用现有模块：

- 文件路径工具：`src/core/filePath.js`
- Tauri 调用包装：`src/core/tauriBridge.js`
- 打开文件：`open_path`
- Word/WPS/LibreOffice 转 PDF 链路：后续复用证据扫描中的转换引擎
- 应用设置和菜单排序：现有模块注册机制

## .docsytpl 格式

`.docsytpl` 是 zip 包，不是一个塞 base64 的 JSON 文件。

```text
template.docsytpl
├── manifest.json
└── template.docx
```

`manifest.json` 保存字段定义和模板元数据；`template.docx` 保存已经写入稳定标记的 Word 模板。

历史数据、候选值和统计学习不写进 `.docsytpl`。这些数据后续存入本地 SQLite，避免模板文件越用越大，也避免分享模板时泄露历史案件信息。

## 字段模型

字段只描述“在哪里、叫什么、是什么类型”。字段不保存前缀、后缀、分隔符等生成逻辑；这些文字应保留在 Word 模板正文里。

```json
{
  "id": "fld_01HABC",
  "name": "法院",
  "label": "审理法院",
  "semantic_key": "法院",
  "type": "text",
  "required": true
}
```

字段可以附加少量渲染属性，但这些属性必须是通用规则，不绑定具体案由或具体角色文字。例如“空值隐藏前置文本/后缀”适用于原告、被告、第三人、律师、代理人等任何可选字段。

### 公共规则库与内部历史库

字段确认阶段可以使用两类数据，但两者必须隔离：

- 公共规则库：随 Docsy 发布，保存法院名称模式、常见法院名种子、最高人民法院公布的案由体系、案号格式、日期格式等公开规则。公共规则只用于“猜测字段类型和字段名”，不写入用户案件数据。
- 内部历史库：保存在本机 SQLite，保存用户自己填过的字段值、同字段常用值、跨字段关联和跨模板关联。内部历史只服务当前用户，不合并进公共规则库。

法院名录与案由库都很小，不会明显增加程序体积。案由规则以最高人民法院发布的《民事案件案由规定》为准；2025 年修改后的案由规定自 2026 年 1 月 1 日施行，共 1055 个案由。第一阶段先内置常用案由和法院名称识别模式，后续再补全全量、可更新的公共数据文件。

日期识别必须覆盖完整日期和空白日期：

- `2026年4月23日`
- `2026年  月  日`
- `年  月  日`
- `年月日`
- `____年__月__日`
- `2026-04-23` / `2026/04/23`

扫描时如果 Word 把日期拆成 `2026`、`年`、`4`、`月`、`23`、`日` 多个黄色 run，前端可以先自动合并为一个候选日期；用户仍可手动拆分。

### 字段类型

字段类型描述“用户要填什么值、用什么控件填、渲染时按什么值处理”。前置文本、后缀、连接词描述“字段周围的固定文本如何跟随字段显示或隐藏”，因此属于字段结构规则，不作为独立字段类型。一个字段可以同时是 `party_list`，并带有空值前置文本/后缀。

| 类型 | 用途 | 模板内表示 | 渲染 |
| --- | --- | --- | --- |
| `text` | 法院、案号、地址等普通文字 | `<w:sdt>` 内容控件 | 替换控件内容 |
| `date` | 日期 | `<w:sdt>` 内容控件 | 格式化后替换 |
| `select` | 案由、角色等单选文本 | `<w:sdt>` 内容控件 | 替换为选中值 |
| `party_list` | 当事人列表 | `<w:sdt>` 内容控件，渲染时可复制所在表格行 | 正文中用顿号连接；表格行中按项复制行 |
| `checkbox` | 单个是否项 | marker 内容控件 | 改勾选符号 |
| `radio_group` | 互斥勾选 | 多个 marker 内容控件 + manifest 组关系 | 只选一个 |
| `checkbox_group` | 多选勾选 | 多个 marker 内容控件 + manifest 组关系 | 可选多个 |

### 当事人与可选字段

`party_list` 在句子中渲染为顿号连接：

- 一个值：`张三`
- 多个值：`张三、李四`

填写时按列表顺序生成，顺序本身是字段值的一部分。每一项可以带自己的后缀：

- 原告列表：`浦项股份有限公司`、`浦项（佛山）钢材加工有限公司` → `浦项股份有限公司、浦项（佛山）钢材加工有限公司`
- 律师列表：`吕晗 + 律师`、`张三 + 实习律师` → `吕晗律师、张三实习律师`

列表前面的角色词通常只出现一次，例如 `原告[原告]`，不是每个原告前都加一次 `原告`。如果模板确实要求每一项都带固定文字，可以把固定文字填入每一项的后缀/前置文本，或拆成单独字段处理。

标点和连接词不属于角色词本身。`，第三人` 应拆成连接文本 `，` 和角色词 `第三人`；但如果第三人字段为空，空值规则要删除完整前置文本 `，第三人`，否则会留下 `被告A公司，之间` 这类病句。同理，`与被告` 中 `与` 是连接文本，`被告` 才是角色词。

`radio_group` / `checkbox_group` 与 `party_list` 的共同点是都包含多个项目，但渲染目标完全不同：

- `party_list` 生成一串文本值，例如 `A公司、B公司`
- `checkbox_group` / `radio_group` 只切换既有选项前面的符号，例如 `□` 变成 `☑`

确认界面应把“方框 + 后面的选项文字”展示为一个选项，便于用户理解；但保存时只把方框符号做成 marker 内容控件，选项文字保留为 Word 原文。这样可以避免破坏选项文字的格式。

只有当字段位于 Word 表格行中时，Docsy 才复制整行，以支持“每个当事人一行”的表格格式。普通段落中的 `party_list` 不复制段落，避免把一整段事实说明复制多遍。

可选字段使用 `optionalRule`：

```json
{
  "type": "party_list",
  "name": "第三人",
  "optional_rule": {
    "enabled": true,
    "remove_empty_prefix": "，第三人",
    "remove_empty_suffix": ""
  }
}
```

当字段为空时，Docsy 删除字段本身，并按配置删除字段周围的固定文字：

- `remove_empty_prefix`：从字段前一个文本节点末尾删除，例如 `原告`、`被告`、`，第三人`
- `remove_empty_suffix`：从字段后一个文本节点开头删除，例如 `律师`、`代理人`

因此“被告甲公司，第三人[字段]之间纠纷”会变成“被告甲公司之间纠纷”；“委托[字段]律师为代理人”在字段为空时会变成“委托为代理人”。这些规则不是身份专用规则，内容由用户在字段确认表格中设置。

同一个字段名可以出现在多个位置，并共用同一个填写值。空值规则有两种作用范围：

- 全部同名：同一个字段名的所有出现位置共用同一组空值前缀/后缀
- 仅此位置：只处理当前这一处出现位置，适合一个字段在正文不同位置分别带有不同前缀或后缀

例如同一个 `lawyer_name` 字段在正文中可能同时出现在“委托[lawyer_name]律师”和落款/表格中。前者可以把 `律师` 设为“仅此位置”的空值后缀，后者不受影响。

### 所函模板元素规则示例

示例文本：

```text
律师事务所函
苏州市中级人民法院：
原告浦项股份有限公司（POSCO CO., Ltd.）、浦项（佛山）钢材加工有限公司与被告上海蔚来汽车有限公司、蔚来汽车销售服务有限公司、苏州蔚然汽车销售服务有限公司，第三人A公司之间侵害发明专利权纠纷一案，原告浦项（佛山）钢材加工有限公司委托吕晗律师为一审阶段的诉讼代理人，特此告知。

北京志霖律师事务所
2026年4月23日
```

推荐拆分：

| 模板片段 | 角色 | 建议字段 |
| --- | --- | --- |
| `律师事务所函` | 固定标题 | 不标字段 |
| `苏州市中级人民法院` | 法院名称 | `法院`，类型 `text`，必填 |
| `：` | 固定标点 | 不标字段 |
| `原告` | 原告列表前置文本 | 不作为字段；如原告可空，设为 `原告` 的空值前置文本 |
| `浦项股份有限公司...、浦项（佛山）钢材加工有限公司` | 原告列表 | `原告`，类型 `party_list` |
| `与被告` | 连接文本 + 被告角色词 | 通常作为固定文本；如被告可空，按模板需要设空值前置文本 |
| `上海蔚来汽车有限公司...` | 被告列表 | `被告`，类型 `party_list` |
| `，第三人` | 连接文本 + 第三人角色词 | 第三人可空时，设为 `第三人` 的空值前置文本 |
| `A公司` | 第三人列表 | `第三人`，类型 `party_list` |
| `之间` | 固定连接词 | 不标字段 |
| `侵害发明专利权纠纷` | 案由 | `案由`，类型 `text` 或 `select` |
| 第二个 `原告` 后的公司名 | 委托人/出函主体 | 可复用 `委托人`，类型 `text` 或 `party_list` |
| `吕晗` | 律师姓名 | `律师`，类型 `text`；后面的 `律师` 是后缀 |
| `律师` | 律师姓名后缀 | 如律师可空，设为 `律师` 的空值后缀 |
| `一审` | 阶段 | `阶段`，类型 `select` 或 `text` |
| `北京志霖律师事务所` | 律所名称 | `律所名称`，类型 `text`，可给默认值 |
| `2026年4月23日` | 日期 | `日期`，类型 `date` |

这类模板的连接词应尽量留在 Word 原文中。字段只表示真正会变化的内容；角色词、标点、称谓后缀通过“空值前置文本/后缀”控制是否随空字段消失。

下划线占位不作为通用规则。用户可以在 Word 里临时用下划线辅助标记，但 Docsy 不应把下划线本身解释为某种字段类型。

## 标黄扫描

用户在 Word 里把可变内容标黄。Docsy 扫描：

- `word/document.xml`
- `word/header*.xml`
- `word/footer*.xml`

扫描结果不是最终字段，只是候选 mark。每个 mark 保留：

- `id`
- `part`
- 原始顺序
- 文本
- 所在段落上下文
- 是否像勾选符号

不要在后端强行把连续黄色 run 合并成一个字段。连续 run 是否属于同一字段、不连续 run 是否共用一个字段，交给 Docsy 确认界面处理。

确认界面支持两类操作：

- **分组**：选中多个 mark 后设置为同一字段名。同一字段名、同一字段类型会合并成一个逻辑字段；对于勾选类型，同名字段会合并为一组。
- **拆分**：一个黄色 mark 内如果包含多个可变片段，可以按字符范围拆成多行。保存模板时 Docsy 会把原 run 切成多个片段，并只把拆出的片段包成内容控件。

字段确认界面应尽量减少用户手工命名：

- 标黄文本本身像法院、案号、案由、日期时，直接推断为对应字段。
- 标黄文本本身不明显时，结合段落上下文推断，例如 `案号：...`、`送达地址：...`、`...律师`、`...阶段`。
- 选中多行后显示就地操作条，用户可以直接合并为字段、设为前置文本或设为后缀，不需要到页面其他位置找按钮。
- 表格下方显示空值规则概览，把所有前置文本、后缀和字段空值处理集中列出，避免用户逐行点设置才知道模板规则。

字段与 mark 的关系使用 `markRefs` 记录：

```json
{
  "mark_refs": [
    { "mark_id": "mark_xxx_7", "start": 0, "end": 2 },
    { "mark_id": "mark_xxx_9", "start": null, "end": null }
  ]
}
```

`start/end` 为空表示整个 mark；有值时表示按 Unicode 字符序号切分，避免中文被 UTF-8 字节位置切坏。

## 稳定定位

扫描阶段可以使用 run 顺序定位。保存为模板时不能长期依赖 run index，因为 Word 重存或用户编辑会改变 run 结构。

保存 `.docsytpl` 时，Docsy 把字段写入稳定标记：

- 普通字段：把黄色 run 包成 `<w:sdt>` 内容控件，`w:tag` 使用字段 id。
- 勾选字段：把单个勾选符号 run 包成极小的 `<w:sdt>` marker，`w:tag` 使用 `字段 id + option id`。

渲染时通过 `w:tag` 查找字段，不通过第几个段落、第几个 run 查找字段。

## 勾选框

法律文书中的勾选框是独立字段类型，不是普通 text。

常见形态：

```text
□ 一般授权    ☑ 特别授权
(  ) 自然人    (√) 法人
☑ 财产保全    □ 证据保全
```

处理原则：

- marker 字符和标签文字分开建模。
- 组关系存在 `manifest.json`。
- 渲染时只改 marker 控件内的 `<w:t>` 文本，不动 `<w:rPr>`。
- 保留原字体，兼容 Wingdings、Symbol、Webdings 等插入符号。
- 如果无法判断选中/未选中符号，由用户在确认界面选择。

## 当前实现范围

已实现的稳定闭环：

1. 扫描 Word 黄色高亮 mark。
2. 确认字段类型：`text`、`date`、`select`、`party_list`、`checkbox`、`radio_group`、`checkbox_group`。
3. 保存 `.docsytpl`。
4. 打开 `.docsytpl` 填表。
5. 生成 docx。
6. 打开生成结果。
7. 生成后记录本地 SQLite 历史。
8. 打开模板时回填上次值。
9. 字段输入时显示同模板常用值、同通用字段名跨模板常用值、当前输入触发的跨字段建议。

当前不做的范围：

- AI 校验
- Word 原生 `w15:repeatingSection` 控件写入。Docsy 采用更稳的渲染期复制：如果 `party_list` 字段所在位置在表格行内，复制整行；否则作为句内列表用顿号连接。这更符合律所模板里“正文句子”和“表格清单”的两种实际用法，也避免修改 Word 命名空间导致兼容问题。

## 多模板材料包

材料包用于一次生成所函、送达地址确认书、授权委托书等多个模板。

设计原则：

- 每个 `.docsytpl` 仍然是独立模板，材料包不把模板打包成另一个大文件。
- 前端按通用字段名（内部字段 `semantic_key`）优先、字段名其次聚合字段，用户只填一次共用信息。通用字段名默认等于字段名，可以手动改为跨模板统一名称。
- 生成时分别调用同一套 `render_docx_template`，让每个模板继续记录自己的历史学习数据。
- 输出目录由用户选择，文件名使用模板名并自动避让重名。

这个设计保证材料包是对现有模板能力的编排，不引入第二套渲染引擎。

后续继续扩展时，优先做多模板材料包和 repeating section，不要把旧系统的字典编辑器、手写 infer 规则和 Docsy 内部选区定位重新引入。

## 历史与建议

历史数据存入本地 SQLite：

```text
template_history.sqlite3
├── generation_runs
└── field_history
```

建议来源：

- `lastValues`：同模板最近一次生成记录。
- `fieldSuggestions`：同模板同字段的高频历史值。
- `semanticSuggestions`：同通用字段名的跨模板高频历史值。
- `associationSuggestions`：当前字段值触发的同次生成关联值，例如填写某法院后提示常见案号片段。

这些建议只辅助填表，永远不自动覆盖用户已输入内容。
