# docx 结构研究笔记

> 用 5 份真实所函 docx 样本逆向，作为 Docsy 模板生成模块的实现依据。
> 样本目录：`templates/`

## 1. docx 顶层结构

docx 实为 zip 容器：

```
xxx.docx
├── [Content_Types].xml          # MIME 注册
├── _rels/.rels                  # 顶层关系
├── docProps/                    # 文档元数据（作者、标题等）
└── word/
    ├── _rels/document.xml.rels  # document 引用关系
    ├── document.xml             # 正文 ★主要操作对象
    ├── styles.xml               # 段落/字符样式定义 ★只读
    ├── settings.xml             # 文档设置
    ├── fontTable.xml            # 字体注册
    ├── theme/                   # 主题
    └── media/                   # 图片（如有）
```

简单所函通常没有 `header*.xml` / `footer*.xml`，但要按存在处理。

## 2. document.xml 段落—Run 模型

```xml
<w:body>
  <w:p>                          <!-- 段落 -->
    <w:pPr>...</w:pPr>           <!-- 段落属性：对齐/行距/缩进/段间距 -->
    <w:r>                        <!-- Run：一段连续同样式的文本 -->
      <w:rPr>...</w:rPr>         <!-- Run 属性：字体/字号/加粗/下划线/颜色 -->
      <w:t>文本</w:t>             <!-- 实际文本节点 -->
    </w:r>
    <w:r>...</w:r>               <!-- 同段落内可有多个 Run -->
  </w:p>
</w:body>
```

**关键事实**：
- 同一段连续文字常被 Word 拆成多个 `<w:r>`，即使样式完全一致。例如"北京知识产权法院"在样本里是 `<w:r>北京知识产权</w:r><w:r>法院</w:r>` 两个 run。
- `<w:t>` 文本可能含 `xml:space="preserve"`，须保留。
- 占位符 `{{key}}` 在用户用 Word 输入时**也会被拆**，因此"占位符落在单 run 内"在自然制作的模板里**不成立**——必须先做 run 合并预处理。

## 3. 当事人下划线的两种风格

样本里发现两套约定：

**风格 A（细粒度）**：只给当事人主体名画下划线
```
原告 [日本制铁株式会社] 与被告 [国家知识产权局]、第三人 [浦项股份有限公司]、[安赛乐米塔尔公司]、[上海蔚来汽车有限公司] 之间...
```
其中 `[]` 标记的是带 `<w:u w:val="single"/>` 的 run。

**风格 B（粗粒度）**：从"再审申请人 + 名字"开始一直划到"案号" 全段下划线
```
[再审申请人 迪思杰（北京）数码技术有限公司与再审被申请人 上海英方软件股份有限公司、李天伟侵害技术秘密纠纷一案，二审案号（2024）京民终1029号、一审案号（2022）京73民初1609号]，
```

**结论**：Docsy 的 party 字段不假设"只下划线姓名"，而是**继承占位符所在 run 的 rPr 整套属性**——用户在 Word 里给整个占位符画下划线，姓名就带下划线；只给姓名子串画下划线，那只有姓名带。这正是设计文档 14.3 节的"样式继承"原则。

## 4. 字体规则（与设计文档 6.3 一致）

| 区段 | 字体 | 字号（w:sz 单位为半点） | 加粗 |
|---|---|---|---|
| 标题"律师事务所函" | 黑体 | 40（=20pt，二号） | 隐式（黑体本身） |
| 法院名 | 仿宋 | 32（=16pt，三号） | `<w:b/>` 加粗 |
| 正文其他 | 仿宋 | 32（三号） | 否 |
| 律所署名 / 日期 | 仿宋 | 32（三号） | `<w:b/>` 加粗 |

字体由 `<w:rFonts w:ascii="仿宋" w:hAnsi="仿宋" w:eastAsia="仿宋" w:cs="仿宋"/>` 声明，需要四个属性都设。

## 5. 占位符替换的实现策略

### 5.1 第一步：run 合并预处理

加载 docx 后，扫描每段，把**相邻、rPr 完全相同**的 `<w:r>` 合并成一个。这样占位符 `{{court}}` 即使在用户的模板里被拆到多 run，合并后也会落到单 run 内。

合并算法：
1. 按 `<w:p>` 切段
2. 段内顺序遍历 `<w:r>`
3. 若当前 run 与前一个 run 的 rPr 相同（XML 比对），合并 `<w:t>` 文本，丢弃当前 `<w:rPr>` 标签
4. rPr 不同则不合并

### 5.2 第二步：扫占位符

合并后，每段拼出纯文本，正则 `\{\{(\w+)\}\}` 找占位符。占位符在拼接文本里的位置 → 反查到具体 `<w:r>` 的 `<w:t>` 中。

需要处理的边界：
- 占位符跨 run 边界：合并后理论上不应再出现；如果还出现说明用户在模板里用了不同样式拼接，按"拒绝标记"处理（设计文档 17 章）。
- `<w:t>` 的 `xml:space="preserve"`：替换文本含首尾空格时必须加。

### 5.3 第三步：按字段类型替换

**text 字段**：直接替换 `<w:t>` 文本内容（XML 转义 `&<>`）。

**party 字段**（多人，顿号分隔）：把单个 `<w:r>{{plaintiffs}}</w:r>` 拆成：
```
<w:r><w:rPr>原 rPr</w:rPr><w:t>姓名1</w:t></w:r>
<w:r><w:rPr>原 rPr 去掉 w:u</w:rPr><w:t>、</w:t></w:r>
<w:r><w:rPr>原 rPr</w:rPr><w:t>姓名2</w:t></w:r>
...
```

**reference 字段**：值确定后等同 text 字段。

**hideable 字段**（值为空时）：把整个 `<w:r>` 删除；如果占位符在段首或独占段，可选删整段 `<w:p>`。

### 5.4 第四步：回写 zip

- 用 `zip` crate，开 `ZipWriter`，对未修改的文件用 `raw_copy_file` 保留原压缩参数和顺序
- 修改过的 `document.xml` / `header*.xml` / `footer*.xml` 用 `start_file` 重新写
- `[Content_Types].xml` 和 `_rels/.rels` 必须出现在前面（zip 偏移敏感）

## 6. 占位符扩展属性建议

基于用户原话"占位符可以加属性，比如会不会在后面直接填上、或者能不能隐藏、最后委托人有没有在前面出现过"——映射到 fields.json：

| 属性 | 含义 | 示例场景 |
|---|---|---|
| `inline_after: true` | 字段后无空格直接拼下一段文本 | 案号"（2026）京73行初XXX号" |
| `hideable: true` | 值为空则隐藏所在 run（或整段） | 第三人为空时整段不出现 |
| `references: ["plaintiffs", ...]` | 候选值来源是另几个字段 | 委托人名字从原告/被告/第三人选 |
| `default_role: "原告"` | party 字段的默认角色 | 原告字段默认带"原告"前缀 |
| `validate_appears_before: ["plaintiffs", "defendants", "third_parties"]` | 校验：值必须在前序字段中出现过 | 委托人姓名必须是当事人之一 |
| `style_inherit: "preserve"` | 替换时严格继承 rPr（默认） | — |
| `multiple: true` + `separator: "、"` | 多值字段的连接符 | party、lawyers |
| `separator_style_drop: ["u"]` | 多值连接符要去掉的样式属性 | 顿号不带下划线 |

## 7. WPS 兼容性发现

样本里出现了 WPS 专属命名空间：
```
xmlns:wpsCustomData="http://www.wps.cn/officeDocument/2013/wpsCustomData"
```

只要不破坏这些命名空间声明、不修改 WPS 的扩展节点，回写后 WPS 仍能正常打开。这进一步支持"只改文本节点、其余原样保留"的策略。
