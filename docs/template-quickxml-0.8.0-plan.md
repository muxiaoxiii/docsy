# Docsy 0.8.0 模板引擎 quick-xml 完整迁移计划

## 定位

0.7.x 主分支继续基于当前模板系统做稳定性、UI、历史数据和规则体验优化。

0.8.0 新开 quick-xml 分支，目标是完整替换模板模块的 OOXML 处理引擎。这个分支不做“正则引擎外面再套中间层”的兼容式改造，而是把模板扫描、字段写入、字段渲染、可选前后缀、列表展开、引用字段等核心处理全部迁到结构化 XML 读写模型。

## 为什么做

当前 0.7.x 已经加了危险结构拦截，能避免静默生成明显损坏或错位的文书。但 Word OOXML 天然包含嵌套结构，正则方案无法长期可靠处理：

- 文本框、页眉页脚、脚注尾注中的段落和 run。
- 已有内容控件、书签、超链接、批注、修订痕迹、域代码穿插在文本中。
- 表格内嵌套表格，或一个表格行里有多个可重复字段。
- Word 自动把同一段文字按字体、下划线、语言、拼写检查边界拆成多个 run。
- 前缀、后缀和字段之间可能跨 run，不能只看单个 `<w:t>`。

0.8.0 的目标不是“支持更多特殊文件”，而是让模板引擎的正确性建立在 XML 树结构上，避免靠字符串位置猜测。

## 分支策略

- 分支名建议：`codex/template-quickxml-0.8`
- 版本目标：`0.8.0`
- 主分支：继续维护 `0.7.x`，只接受当前引擎上的稳定性修复和体验优化。
- quick-xml 分支：允许重写 `src-tauri/src/docx_template.rs` 内部结构，但必须保持前端字段模型和 `.docsytpl` manifest 的业务语义尽量稳定。
- 不要求兼容 0.7.x 内部实现细节；如果 0.7.x 模板文件需要迁移，做显式 manifest migration，不做隐式猜测。

## 迁移原则

1. 以 WordprocessingML 结构为准，不以文本正则为准。
2. 保留未触碰 XML 的原始结构和属性，尤其是字体、字号、加粗、斜体、下划线、段落、表格、页眉页脚。
3. 字段替换只改目标文本节点或目标结构，不重建无关 run。
4. 前缀、后缀、连接符、引用和列表项都作为结构化节点范围处理，不再靠字符串 `starts_with` / `ends_with`。
5. 遇到不能安全处理的结构，给出可理解错误，不输出不确定文书。
6. 先保证 `.docx`，`.doc` 仍走转换链；转换后的 `.docx` 再进入同一 quick-xml 管线。

## 新引擎范围

### 1. Package 读写

负责读取和写回 docx zip 包：

- 读取 `word/document.xml`
- 读取 `word/header*.xml`
- 读取 `word/footer*.xml`
- 后续可扩展 `word/footnotes.xml`、`word/endnotes.xml`
- 保留所有非目标 zip entry 原样写回
- 继续做 zip entry 数量和大小限制
- 继续校验 `.docsytpl` manifest 版本和字段类型

### 2. OOXML 树模型

用 quick-xml 解析目标 XML part，建立最小可编辑树：

- Element：名称、属性、children
- Text：文本内容
- Raw/Other：可保留但不主动解释的节点

不做完整 OpenXML SDK 级别模型，只做 Docsy 模板引擎需要的结构化树。

### 3. 文本索引

为每个 XML part 建立稳定索引：

- paragraph id
- run id
- text node id
- 所属表格行 id
- 所属 sdt id
- 所属 hyperlink/bookmark/revision 等上下文
- 高亮、加粗、斜体、下划线等基础格式

扫描阶段和写入阶段必须使用同一个索引来源，彻底取消“扫描按段落数 run，写入按全文数 run”的隐式前提。

### 4. 标黄扫描

扫描所有黄色高亮文本节点，输出给前端确认：

- 连续 run 不强行合并为一个字段，但要在候选里标出相邻关系，方便 UI 合并。
- 同一字段跨多个 run 时，保存为多个结构化 text node reference。
- 勾选框 run 保留原始字体和符号文本。
- 表格、页眉、页脚、文本框中的标黄都应能被扫描到。

### 5. 保存模板

保存时把已确认字段写入 docx：

- 文本、日期、列表、引用字段写为 Docsy 自己的内容控件或等价结构。
- 删除文本字段直接删除目标 text range。
- 前缀、后缀、连接符建立结构化归属关系，记录在 manifest，同时在 docx 中保留原文字。
- 勾选组不强行包 sdt；记录 marker text node reference，渲染时只改该 text node 的值。
- 禁用字段移除黄色高亮但保留原文。
- 非黄色高亮必须原样保留。

### 6. 渲染模板

渲染时按结构化字段处理：

- 文本字段：替换对应内容控件内文本，保留 run 属性。
- 日期字段：输入归一化后按字段格式输出。
- 当事人列表：按列表项渲染，连接符由列表规则产生，不残留孤立顿号。
- 可重复后缀：每个列表项独立处理后缀。
- 非重复前后缀：字段有值时显示一次，字段空值时整体删除。
- 引用字段：只读取选定来源值并复制到引用位置，不反向修改来源字段。
- 勾选组：互斥组只选一个，多选组按数组值设置。
- 删除文本：删除目标范围，不影响周边格式。

### 7. 表格重复

重新定义表格列表展开规则：

- 单个列表字段占一行：按项复制整行。
- 同一行多个列表字段：默认拒绝；除非字段显式设置为 zip 对齐。
- 嵌套表格：结构化识别后再决定是否支持；不能安全复制时拒绝。
- 行复制必须复制完整 `<w:tr>` 节点，不得用正则找第一个 `</w:tr>`。

### 8. 可选前后缀和连接符

0.8.0 要把这部分作为重点：

- 前缀/后缀可以跨多个 run。
- 连接符只处理同一列表中相邻项之间的顿号等连接符。
- 逗号、句号、括号不默认当作列表连接符删除。
- 字段为空时删除其绑定的前缀、后缀和必要连接符。
- 字段有值时可按填写页结构设置覆盖前后缀。

### 9. `.docsytpl` manifest

manifest 继续作为业务数据主结构，但需要加版本：

- `formatVersion: 2`
- 字段引用从 mark id 升级为结构化 node reference。
- 保存来源 part、node path、text range、format snapshot。
- 旧模板迁移必须显式执行，迁移失败要提示重新导入 Word 模板。

## 建议模块划分

Rust 侧建议拆成：

- `docx_template/package.rs`：docx/docsytpl zip 读写与安全限制
- `docx_template/ooxml.rs`：quick-xml 树结构读写
- `docx_template/index.rs`：文本索引、段落/run/text node 定位
- `docx_template/scan.rs`：标黄扫描和候选输出
- `docx_template/save.rs`：字段写入和 manifest 生成
- `docx_template/render.rs`：模板渲染
- `docx_template/rules.rs`：日期、列表、前后缀、引用、勾选组规则
- `docx_template/migration.rs`：manifest 版本迁移

这不是“中间层包旧逻辑”，而是新的引擎内部职责拆分。旧 `docx_template.rs` 应逐步缩成命令入口和类型导出。

## 验收用例

0.8.0 分支至少覆盖这些真实场景：

1. 所函：原告、被告、第三人、案由、案号、律师、诉讼阶段、律所、日期。
2. 同一当事人前文出现一次，后文作为引用出现一次，引用修改不反向影响来源。
3. 一个字段跨多个 run，包含下划线、加粗、斜体，生成后格式保留。
4. 日期只标黄年月日中的数字，仍能作为一个日期字段处理。
5. 勾选框互斥组选项，保留 Wingdings/Symbol 字体。
6. 当事人列表只有一项时不出现顿号，多项时按顿号连接。
7. 空第三人时删除“第三人”前缀和相关语法结构，不删除无关逗号。
8. 页眉页脚中的模板字段可扫描、保存、渲染。
9. 表格行单列表字段可按项复制，同行多列表字段明确拒绝或 zip 对齐。
10. 文本框中字段可扫描并生成，不发生 run index 错位。
11. 既有 Word 内容控件不被破坏；若无法嵌套处理，给出明确错误。
12. 非黄色高亮、批注、书签、超链接不因保存模板丢失。

## 测试策略

- 单元测试：XML 片段级扫描、保存、渲染。
- 回归测试：用最小 docx zip 包验证 package 写回。
- 端到端测试：导入 Word → 保存 `.docsytpl` → 填写 → 生成 docx → 读取 document.xml 验证文本和结构。
- 视觉抽检：核心模板用 Word/LibreOffice 打开验证格式。
- 安全测试：超大 zip entry、缺 manifest、未知 manifest 版本、未知字段类型。

## 风险和取舍

- quick-xml 迁移会提高正确性，但会引入较大开发量，不能和 0.7.x 小修混在一起。
- 0.8.0 分支可以接受短期功能不完整，但不能接受静默生成错误文书。
- 主分支继续用当前防御版正则引擎，保持可用和可打包。
- 如果 0.8.0 的结构化引擎没有覆盖某个 Word 特性，应明确拒绝该模板，而不是回退到正则。

## 里程碑

### M1：基础树和索引

- 引入 quick-xml。
- 完成 docx package 读写。
- 完成 document/header/footer XML 树解析和写回。
- 完成 text node/run/paragraph/table row 索引。

### M2：扫描与保存

- 标黄扫描切到 quick-xml。
- 保存模板切到结构化写入。
- 删除黄色高亮保留其他格式。
- 生成 manifest v2。

### M3：渲染

- 文本、日期、引用、删除文本、勾选组渲染完成。
- 前后缀和连接符按结构化范围处理。
- 当事人列表 inline 渲染完成。

### M4：表格与复杂结构

- 单列表表格行复制。
- 同行多列表字段检测和策略。
- 文本框、页眉、页脚稳定覆盖。

### M5：迁移与验收

- 旧 manifest 显式迁移或重新导入提示。
- 完成真实所函模板验收。
- 完成测试矩阵和文档更新。
