# PDF 证据处理整体设计

更新时间：2026-07-13

## 目标边界

Docsy 的 PDF 模块不是通用 PDF 编辑器，而是面向法律证据材料的本地处理工具。核心目标是把平时分散维护的证据 PDF，稳定地整理为可提交、可分享、可拆回的证据文件。

主线工作流：

1. 导入一组独立证据 PDF，例如 `证据1.pdf`、`证据2.pdf`。
2. 识别或设置每个 PDF 的证据身份，例如 `证据1`、`证据2`、文件名、自定义名称、重新编号。
3. 读取、删除、编辑或插入页眉页脚文本。
4. 按最终合并顺序计算连续页码，例如 `1/200`、`第1页，共200页`。
5. 可选统一页面格式，例如规范为 A4、删除批注。
6. 输出每个证据的处理后副本，并可选输出一个合并 PDF。
7. 对已经合并的 PDF，可以根据页眉、页码模式或手动页段拆分回证据文件列表。

非目标：

- 不做完整的 Acrobat 替代品。
- 不以破坏原始 PDF 为代价修改文件；默认输出副本。
- 不把模板、文书生成、字典等已移除模块重新引入 PDF 工具。

## 法律证据场景

证据整理的实际使用方式决定了模块结构：

- 平时维护时，证据应保持为多个独立 PDF，便于替换、检查、补充。
- 提交或分享时，通常需要合并为一个 PDF，并要求页码在合并后连续。
- 页眉一般标识证据身份，常见来源包括文件名、序号、中文序号、自定义表格名称。
- 页脚一般是页码，常见格式包括纯数字、`1/200`、`第1页`、`第1页，共200页`。
- 现有 PDF 可能已经由 Acrobat、WPS 或其他软件插入页眉页脚；Docsy 需要先读取和处理已有内容，避免新旧文字重叠。
- 合并后的 PDF 也可能需要拆回多个证据文件；拆分依据优先来自页眉，其次来自页码模式和人工调整。

因此，PDF 模块的核心对象不是“某个页面上的一段文字”，而是“证据文件列表”和“每个证据对应的页码范围、页眉、页脚、输出策略”。

## 统一数据模型

前后端都围绕同一份证据清单工作。

```ts
type EvidencePdfItem = {
  id: string
  sourcePath: string
  originalPath?: string
  displayName: string
  evidenceLabel: string
  order: number
  pageCount: number
  pageStart: number
  pageEnd: number
  detectedHeader?: HeaderFooterDetection
  detectedFooter?: HeaderFooterDetection
  outputPath?: string
  status: "ready" | "warning" | "processing" | "done" | "failed"
}

type HeaderFooterDetection = {
  text: string
  normalizedText: string
  region: "header" | "footer"
  pageRange: { start: number; end: number }
  confidence: number
  source: "annotation" | "content-text" | "visual-region" | "manual"
}

type EvidencePdfSession = {
  items: EvidencePdfItem[]
  totalPages: number
  headerRule: HeaderRule
  footerRule: FooterRule
  cleanupRule: CleanupRule
  pageFormatRule: PageFormatRule
  annotationRule: AnnotationRule
  outputRule: OutputRule
}
```

合并和拆分必须共用这套模型：

- 导入独立 PDF 时，`items` 来自文件列表。
- 导入合并 PDF 时，`items` 由读取结果或人工页段拆分生成。
- 插入页眉时，方向是 `items -> header preview -> output`。
- 删除或读取页眉时，方向是 `PDF pages -> detected headers -> items`。
- 拆分时，方向是 `merged PDF -> detected page ranges -> items -> output files`。

## 页眉页脚读取

读取分三层进行，不能只依赖一种方式。

### DocDeck 复盘结论

已有 DocDeck 项目中做过多种页眉页脚试验，结论应沉淀到 Docsy 的实现约束中，而不是直接搬代码。

可复用的判断：

- PDF 中可以存在标准标记内容，典型形式是 `/Artifact << /Type /Pagination /Subtype /Header >> BDC ... EMC` 或 `/Subtype /Footer`。
- 标准标记内容可能直接包含文本，也可能通过 `/XObject ... Do` 引用 Form XObject；读取时要同时检查页面内容流和被引用的 XObject 流。
- 如果不存在 Artifact 标记，仍可以通过页面顶部/底部区域、重复出现次数、字体大小、文本模式来识别候选页眉页脚。
- 候选识别需要输出结构化结果，包括文本、页码范围、区域、bbox、重复次数、是否页码模式、置信度。

必须避免的实现方式：

- 不直接把页面 `/Contents` 合并为一个解码后的字符串再写回。这样容易破坏原有压缩流、过滤器、资源引用、图形状态和对象结构。
- 不用正则直接删除内容流片段后保存为唯一内容流，除非先通过解析器确认 BDC/EMC 成对、图形状态平衡、删除范围不跨越其他内容。
- 不把 PyMuPDF 的 `insert_text`、`insert_textbox` 或 text annotation 误称为标准页眉页脚；它们可能能显示文字，但不等于可编辑的 Header/Footer 语义对象。
- 不手工拼接中文 Type0/CID 字体对象。中文字体应由稳定库生成或由现有 PDF/字体载体复制，并在输出后验证 ToUnicode 和渲染结果。
- 不在 A4 规范化时直接改 MediaBox/CropBox 后用字符串包裹原内容流。旋转页、继承资源、内容流数组和图形状态都可能因此损坏。

Docsy 的策略：

- 检测/分析阶段可以读取内容流、Artifact、XObject、注释和文本块，但必须只读。
- 删除阶段按风险分级：先删除明确可识别的对象或完整 Artifact 段；无法确认时只做预览和用户确认；最后才使用视觉区域覆盖。
- 写入阶段默认继续使用“独立 overlay PDF + qpdf 合成”，避免手写修改原页面内容流。`pdf-syntax` 这类解析库只用于读和定位标准结构，不作为第一阶段的写入引擎。
- 每次引入内容流编辑能力，都必须配套 `qpdf --check`、渲染对比和至少一个 Acrobat/WPS 来源样本测试。

### PDF 底层库取舍

当前目标不是“去 qpdf”，而是减少脆弱逻辑。qpdf 已经稳定覆盖解锁、合并、拆分、页面信息、overlay 合成和结构校验。把这些全部换成 Rust 库不会让当前实现更简单，反而会让 Docsy 自己承担 PDF 写入、对象编号、xref、object stream、资源继承和字体嵌入的风险。

因此底层库按职责分层：

- qpdf 继续作为安全改写和结构校验底座，尤其是合并、拆分、解锁、overlay、`--check`。
- PDF.js 负责前端真实页面预览和交互坐标。
- Poppler 工具链继续作为后端渲染、文本 bbox 检测和测试验证兜底。
- `pdf-syntax` 可以引入，但只用于解析标准结构和内容流，例如识别 Artifact Header/Footer、BDC/EMC 标记、XObject 引用。它不替代 qpdf 的写入职责。
- `lopdf` 或 qpdf Rust binding 只在能明确减少现有命令封装复杂度时再引入。qpdf Rust binding 本质仍依赖 QPDF 原生库，解决的是调用方式，不是 PDF 复杂度本身。

结论：短期最简单、最稳的是 qpdf 主写入 + PDF.js/Poppler 预览检测 + `pdf-syntax` 只读解析。完全换掉 qpdf 不是当前主线。

### 运行时成本约束

保留 qpdf、PDF.js、Poppler 和可选解析库，不等于把 Docsy 做成重型 PDF 套件。Rust/Tauri 的优势仍然要保留：主程序轻、启动快、非 PDF 功能不被 PDF 工具链拖慢。

后续实现遵守以下约束：

- PDF 工具全部按需调用。应用启动、普通文档功能、非 PDF 页面不初始化 qpdf、Poppler 或 PDF.js worker。
- qpdf、Poppler 作为短生命周期子进程使用，不做常驻服务。批处理时复用任务队列和临时目录，但不让外部进程长期占用内存。
- PDF.js 只在预览组件挂载后加载，离开 PDF 工作台后释放文档、canvas、worker 和缩略图缓存。
- `pdf-syntax` 这类 Rust 依赖只承担轻量只读解析，不把大 PDF 全量常驻内存；优先流式、分页、限定页数扫描。
- 后端检测先使用抽样页和页眉页脚区域，只有用户确认需要深度分析时才扫描全文档。
- 打包策略区分“内置必需”和“可发现外部工具”。如果 qpdf/Poppler 已在系统路径中存在，优先复用；内置工具只作为降低安装门槛的选择。
- 每增加一个外部工具或大型前端依赖，都要记录用途、触发时机、是否可选、失败后的降级路径。
- 性能验收要覆盖启动时间、PDF 工作台首次打开时间、单个 2000 页 PDF 的预览内存、批量处理后的内存回收。

### 超大法律证据包约束

法律证据 PDF 不能按普通办公 PDF 估算。Docsy 的 PDF 工作台必须把“多个扫描件合并后约 2000 页”作为常规上限设计，而不是异常情况。扫描页可能包含高分辨率图片，单页内存成本远高于文字 PDF。

大文件处理遵守以下约束：

- 前端不得默认把超大 PDF 整份读入 JS 内存。普通预览默认走后端单页渲染；`readFile -> getDocument(data)` 仅作为 PDF.js 备用路径或后续小文件交互路径。
- 预览必须虚拟化。只渲染当前页、相邻少量页面和有限缩略图；滚动列表不得一次创建 2000 个 canvas、文本层或注释层。
- 页眉页脚检测默认抽样：首页、末页、每个文件边界页、固定间隔页、疑似变化页。只有用户选择深度检测时，才后台扫描全部页面。
- 批量删除旧页眉页脚时，候选识别和删除计划分开执行。先生成页段、区域、文本模式和风险等级，再让用户确认；不要边扫边不可逆写入。
- A4 规范化是高成本操作，必须按文件或页段分块执行，支持跳过已是 A4 的页面，支持暂停、取消、失败重试和保留中间结果。
- 合并、拆分和插入页眉页脚必须走文件流/临时文件管线，不把合并后的完整 PDF 或全部页面图像放进内存。
- 外部工具并发必须受控。默认同一时间只运行 1 个重型 PDF 写入任务；渲染/检测可低并发，但必须有队列、取消和超时。
- 每个长任务都要有进度事件，至少包含当前文件、当前页段、总页数、已完成页数、当前阶段和可恢复的输出路径。
- 扫描件不默认 OCR。页眉页脚检测先处理 PDF 已有文本层和标准结构；如果页面只有图片，默认使用位置覆盖/插入策略，OCR 作为未来可选能力。
- 任何一次批处理结束后都要释放 PDF.js 文档、canvas、后端临时图片、检测缓存和子进程句柄，避免连续处理多个大文件后内存累积。

### 标准对象层

优先读取 PDF 中可区分的对象：

- 注释、批注、文本标注、自由文本框。
- 由编辑软件保留的可编辑页眉页脚对象。
- 页面资源中重复出现且位置稳定的文本对象。

注意：PDF 标准本身不保证存在“页眉对象”或“页脚对象”。Acrobat/WPS 显示为可编辑页眉页脚，不等于底层一定有统一语义标签。因此标准对象层只能作为优先策略，不能作为唯一策略。

### 内容文本层

如果页眉页脚已经写入页面内容流，应按页面区域和文本模式识别：

- 页眉区域：页面顶部固定高度，默认 15-25 mm，可配置。
- 页脚区域：页面底部固定高度，默认 15-25 mm，可配置。
- 同一文件多页页眉通常相同，或仅包含规律性变化。
- 页脚通常是页码模式，允许页号变化但格式稳定。

文本归一化规则：

- 去除多余空格。
- 统一全角/半角数字。
- 识别中文页码表达。
- 将页码变量抽象为 `{page}`、总页数抽象为 `{total}`。

### 视觉区域层

如果无法可靠提取文本，仍可通过固定区域进行处理：

- 预览页眉页脚所在区域。
- 允许用户框选或调整顶部/底部清除区域。
- 输出时以白底覆盖该区域，再插入新的文本。

视觉区域层不是语义编辑，但在扫描件、扁平化 PDF、不同软件写入方式不一致时最稳定。

## 页眉页脚编辑和删除

编辑和删除需要按来源分级处理。

| 来源 | 读取方式 | 删除方式 | 编辑方式 |
| --- | --- | --- | --- |
| 可区分注释或自由文本 | PDF 对象读取 | 删除对应对象 | 修改对象或删除后重建 |
| 标准 Artifact Header/Footer | 内容流标记读取 | 删除完整 BDC/EMC 标记段 | 删除后重建 |
| 页面内容流中的普通文本 | 内容流文本定位 | 删除/重写内容流，或区域覆盖 | 优先重写；失败则覆盖后重建 |
| 扁平化或扫描件 | 视觉区域 | 区域覆盖 | 覆盖后重建 |

当前稳定实现可以先覆盖区域，但最终架构必须保留三层处理入口，不能把“白底覆盖”写死成唯一删除策略。

删除规则需要支持：

- 删除页眉。
- 删除页脚。
- 删除页码但保留页眉。
- 删除页眉但保留页码。
- 按页码范围删除。
- 按检测到的相同页眉分组删除。

编辑规则需要支持：

- 将检测到的页眉回填到证据列表。
- 在列表中批量改名或重新编号。
- 将列表中的证据名称重新写入页眉。
- 修改页脚页码格式并重新计算连续页码。

## 页码连续计算

页码永远基于当前证据列表的最终顺序计算，而不是每个 PDF 独立计算。

```text
证据1.pdf  12 页  -> 1-12
证据2.pdf   8 页  -> 13-20
证据3.pdf 180 页  -> 21-200
```

页脚模板支持：

- `{page}`
- `{total}`
- `{page}/{total}`
- `第{page}页`
- `第{page}页，共{total}页`

如果用户选择输出每个证据的处理后副本，每个副本仍使用全局页码。例如 `证据2.pdf` 的第一页页脚是 `13/200`，不是 `1/8`。

## A4 规范化

页面规范化是证据输出规则的一部分，不是页眉页脚功能的一部分。

需要支持：

- 检测每页尺寸。
- 标记非 A4 页面。
- 标记横向页面。
- 可选统一到 A4。
- 保留原内容比例，居中放置。
- 支持纵向、横向、自动方向。

实现策略分两类：

| 策略 | 优点 | 缺点 | 使用场景 |
| --- | --- | --- | --- |
| 内容流矩阵缩放 | 保留文本和矢量内容 | 实现复杂，需要处理旋转、裁切框、资源继承 | 可编辑 PDF、正式长期目标 |
| 页面渲染重建 | 稳定、可预览、兼容扫描件 | 牺牲文本可选中性和部分矢量质量 | 扫描件、混合来源 PDF、快速稳定输出 |

Docsy 的架构应同时容纳两者。默认可以给用户明确选项：

- `保留文本优先`
- `稳定版式优先`

如果当前只能稳定完成重渲染，也必须在 UI 和文档中标明其影响，不能伪装为无损编辑。

## 批注删除

批注删除应独立于页眉页脚。

目标支持：

- 删除评论。
- 删除高亮。
- 删除下划线。
- 删除删除线。
- 删除自由文本批注。
- 删除图章或形状批注。

处理原则：

- 优先删除 PDF annotation 对象。
- 不删除页面正文内容。
- 如果批注已被扁平化进内容流，则不能按批注对象删除，只能进入视觉区域或人工处理。

## 合并输出

合并不是单独按钮逻辑，而是输出规则的一部分。

输出选项：

- 只保存每个证据的处理后副本。
- 保存处理后副本，并合并为一个 PDF。
- 只输出合并 PDF。

合并前的处理顺序固定：

1. 读取页面信息。
2. 计算证据列表和全局页码。
3. 可选删除批注。
4. 可选页面规范化。
5. 可选删除已有页眉页脚。
6. 插入或更新页眉页脚。
7. 保存每个证据副本。
8. 按列表顺序合并。

任何步骤失败都应定位到具体文件，不应让整个任务静默失败。

## 拆分输入

导入一个已合并 PDF 时，拆分按以下优先级识别页段。

### 页眉识别

如果页眉存在且稳定：

- 连续页面页眉相同，视为同一个证据。
- 页眉变化处，视为新证据开始。
- 页眉文本可作为默认文件名。

例如：

```text
1-12    证据1
13-20   证据2
21-200  证据3
```

### 页码识别

页脚页码用于辅助判断：

- 验证页面连续性。
- 判断是否存在缺页、重复页、插页。
- 用户选择拆分后保留页码或删除页码。

页码不能单独确定证据边界，除非用户提供页段规则。

### 人工页段调整

如果没有可靠页眉：

- UI 提供页面预览。
- 用户设置 `起始页-结束页`。
- 可以批量生成 `文件1`、`文件2`，其中“文件”前缀可统一修改。
- 如果检测到页眉，可以预填文件名为 `[页眉文字]-文件1`。

拆分输出选项：

- 保留页眉。
- 删除页眉。
- 保留页码。
- 删除页码。
- 重新编号页码。

## 预览模块

预览是核心基础设施，不是某个按钮的附属功能。Docsy 不应自行实现完整 PDF 渲染器；交互预览应优先基于成熟 PDF 预览库封装。

预览实现原则：

- 预览底图优先按场景选择：大文件和默认工作台预览走 Rust 后端 `pdftoppm` 单页渲染，小文件增强交互可走 PDF.js / `pdfjs-dist`。
- PDF.js 负责后续增强的 canvas 渲染、缩放、滚动、旋转、文本层、注释层和页面坐标换算基础。
- Docsy 在页面底图之上叠加自己的规则预览层，包括页眉、页脚、清除区域、拆分页段标记。
- Rust 后端保留 `pdftoppm` 渲染能力，作为默认单页预览、批量任务缩略图、PDF.js 不可用时的兜底、以及 A4 重渲染规范化的底层能力。
- 最终 PDF 写入仍由 Rust 后端执行，不能把 PDF.js 预览层当作真实文件修改结果。

推荐封装方式：

```text

│   └── PdfJsPreview.vue       # PDF.js 预览底座
├── composables/

```

`PdfJsPreview.vue` 对外只暴露 Docsy 需要的业务事件和坐标，不把 PDF.js API 散落到业务组件中。

预览能力：

- 渲染任意 PDF 的任意页。
- 显示页面真实尺寸和 A4 目标尺寸。
- 显示页眉清除区域。
- 显示页脚清除区域。
- 显示将要插入的页眉文本。
- 显示将要插入的页脚页码。
- 支持页码跳转。
- 支持在拆分模式中选择页段。
- 支持在检测模式中高亮识别到的页眉页脚候选。

预览坐标应统一使用 PDF 点位作为内部坐标：

```text
1 inch = 72 pt
1 mm = 72 / 25.4 pt
PDF 坐标原点在左下角
前端显示坐标按渲染图比例映射
```

任何插入位置都必须能从预览坐标反推到 PDF 坐标，避免“预览看起来对，输出位置偏”的问题。

PDF.js 接入边界：

- 可以使用 PDF.js 的 canvas layer 做页面底图。
- 可以使用 text layer 辅助文字选择和页眉页脚候选定位。
- 可以使用 annotation layer 辅助显示和识别批注。
- 不依赖 PDF.js 修改 PDF 文件本体。
- 不直接魔改 PDF.js 源码；如需定制，优先写封装层和 overlay layer。只有在 PDF.js Viewer UI 本身无法满足交互时，才 fork 或 patch viewer 组件。
- 本地文件加载应通过 Tauri 文件读取或受控 asset URL 完成，避免在前端随意暴露任意路径。

后续开发必须遵守：

1. 任何功能都要落到 `EvidencePdfSession` 数据模型上。
2. 页眉页脚处理不能只从“插入”视角设计，必须同时支持读取、删除、编辑、回填列表。
3. 合并和拆分必须共用同一套证据列表和页码范围模型。
4. 预览模块必须是共享基础设施，不能只服务某个页签。
5. 所有输出默认另存副本，不覆盖原文件。
6. 能语义删除时优先语义删除，不能语义删除时才使用区域覆盖。
7. 对会损失文本可选中性的操作，UI 必须明确提示。
8. 旧的临时 overlay 接口只能作为兼容层，不能继续扩大。

## 实施顺序

这不是分版计划，而是依赖顺序。后面的能力必须建立在前面的统一模型上。

1. 建立 `EvidencePdfSession`、页面信息、预览坐标、输出规则。
2. 把当前页眉页脚工作台拆成独立工作台组件。
3. 把后端 `overlay.rs` 拆成 `preview`、`header_footer`、`normalize`、`page_info`。
4. 引入 PDF.js 预览底座，封装 `PdfJsPreview.vue` 和坐标转换层。
5. 将前端调用从 `overlay_pdf_text` 迁移到业务命令。
6. 实现页眉页脚检测，并把检测结果回填到文件列表。
7. 实现对象层批注删除。
8. 实现对象层或内容流层页眉页脚编辑；无法编辑时使用区域覆盖。
9. 实现合并 PDF 的自动拆分建议和人工页段编辑。
10. 统一输出：单文件副本、合并 PDF、拆分 PDF 都通过同一输出规则执行。

## 验收标准

功能完成时，应满足以下场景：

1. 导入 10 个独立证据 PDF，按文件名生成右上角页眉，输出合并 PDF，页脚从 `1/总页数` 连续到最后一页。
2. 导入已有 Acrobat/WPS 页眉页脚的 PDF，能够读取或至少定位页眉页脚区域，删除旧文本后重新插入，输出不重叠。
3. 导入非 A4 PDF，能预览规范化效果并输出 A4 文件。
4. 导入带评论、高亮、下划线的 PDF，能删除批注且不删除正文。
5. 导入一个合并 PDF，能按检测到的页眉生成证据列表和页码范围。
6. 无法自动拆分时，能通过预览人工设置页段并输出多个文件。
7. 拆分时可选择保留或删除原页码。
8. 所有处理均不覆盖原始文件。
